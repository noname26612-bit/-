## 🟠 Важное (закрыть до запуска или сразу после)

### В1. Загрузка файла буферизует всё тело в память ДО проверки размера — DoS памяти на 1.9 ГБ VPS
- **Где:** `src/app/api/tasks/[id]/attachments/route.ts:21,25`; нет лимита в `Caddyfile`, `next.config.ts`.
- **Что не так:** хендлер делает `await req.formData()` + `Buffer.from(await file.arrayBuffer())` — материализует **весь** загружаемый файл в RAM ещё до проверки. Лимит 15 МБ (`validateUpload`) применяется **после** буферизации, по `bytes.byteLength` уже считанного буфера. У Next.js Route Handlers нет дефолтного лимита тела; в Caddy нет `request_body max_size`. Любой аутентифицированный пользователь может прислать multipart-тело в несколько ГБ → Node выделяет буфер на весь размер → OOM/своп → перезапуск контейнера. На app нет `mem_limit` — OOM-killer может задеть и Postgres. Клиентское сжатие (`image-compress.ts`) обходится прямым POST.
- **Доказательство:** `const bytes = Buffer.from(await file.arrayBuffer()); … sizeBytes: bytes.byteLength` — проверка размера внутри `addAttachment`, т.е. после чтения.
- **Рекомендация:** (1) отбивать по заголовку `Content-Length` > 15 МБ → 413 **до** чтения тела; (2) основная мера — `request_body { max_size 16MB }` в `Caddyfile` (режет на уровне прокси); (3) задать `app.mem_limit` (~1200m) в compose, чтобы OOM бил по app, а не по Postgres.
- **Уверенность:** высокая.

### В2. Дата смены берётся из тела клиента — обход штрафа «поздно открыл смену» и порча учёта
- **Где:** `src/domain/shift-service.ts:79` (`openShift(driverId, today)`).
- **Что не так:** `Shift.date` пишется из клиентского `today` (формируется из локальной зоны телефона, `task-ui.ts`), `openedAt` — из серверного времени. Детектор `SHIFT_LATE` и учёт простоя выбирают смены по `date`. Водитель (или сбитая зона устройства, или подделанный запрос) может прислать `today=вчера`: смена запишется на вчера, ночной cron за «сегодня» её не найдёт, за «вчера» уже отработал и не повторяется → штраф `SHIFT_LATE` **не создаётся никогда**; простой в Сводке считается не в том окне.
- **Доказательство:** `openShift` принимает `today` из тела; cron ищет `where:{date:scheduledDay}` по серверному «сегодня».
- **Рекомендация:** вычислять день смены на сервере (`dateKeyInTz(new Date(), KPI_TZ)`), клиентский `today` игнорировать или отвергать при несовпадении с серверным днём МСК. Открытие смены и так online-only — серверное время доступно.
- **Уверенность:** средняя→высокая (вектор подтверждён кодом; реалистичность эксплуатации — средняя).

### В3. Переназначение активной задачи обходит инвариант «одна активная задача»
- **Где:** `src/domain/task-service.ts` — `assignTask` (~507), `planTask` (~575) vs проверка в `transitionTask:667`.
- **Что не так:** `ACTIVE_TASK_EXISTS` проверяется **только** в `transitionTask` при переходе → `IN_PROGRESS`. `assignTask`/`planTask` сохраняют текущий статус и меняют только `assigneeId` (блокируют лишь DONE/CANCELLED). Диспетчер, переназначив задачу в статусе `IN_PROGRESS` на водителя, у которого уже есть «В работе», создаёт у него **две активные задачи** — инвариант (ARCH §5, PRD §5, этап B) нарушается в обход. KPI/полоса отработанного считаются неверно.
- **Рекомендация:** в `assignTask`/`planTask` при смене исполнителя у задачи в `IN_PROGRESS` проверять отсутствие другой `IN_PROGRESS` у нового исполнителя (как в `transitionTask`), либо запрещать переназначение активных задач (требовать сперва `ON_HOLD`). Добавить тест.
- **Уверенность:** высокая. *(Доступно только диспетчеру/админу, частота низкая — но это обход серверного инварианта.)*

### В4. Команда запуска прод-стека не зафиксирована; закрытость порта Postgres зависит от недокументированного слияния compose-файлов
- **Где:** `docker-compose.prod.yml`, `docker-compose.yml`, скилл `deploy-release`; `deploy/README.md` (на который ссылаются конфиги) **отсутствует**.
- **Что не так:** прод-файл в одиночку безопасен (у postgres нет секции `ports`). Но коммит ca9cd1d и память говорят, что на проде dev-файл **сливается** рядом с прод-файлом, а compose конкатенирует массивы `ports` (поэтому в dev-файл добавили bind `127.0.0.1:5432`). Точная команда слияния (`-f docker-compose.yml -f docker-compose.prod.yml`) не записана **нигде**. Авто-override compose подхватывает `docker-compose.override.yml`, а не `.prod.yml`. Итог: закрытость БД-порта держится на устной договорённости. Запуск по документированной команде (только прод-файл) → фикс bind в dev-файле мёртв; запуск со старым dev-файлом без bind → снова утечка `0.0.0.0:5432`.
- **Рекомендация:** не публиковать порт Postgres в проде вообще (для отладки — `docker exec`/туннель); зафиксировать единственную каноничную команду запуска в скилле `deploy-release` и в шапке прод-файла; после деплоя проверять `ss -tlnp | grep 5432`. Создать отсутствующий `deploy/README.md`.
- **Уверенность:** высокая.

### В5. Off-site бэкап выключен по умолчанию и нигде не провязан
- **Где:** `deploy/backup.sh:84`; нет `deploy/cron.d/vanmark-backup` в репо; скилл `deploy-release` (§«Бэкапы») содержит дореформенные ручные cron-строки и не ссылается на новый `backup.sh`; `.env.prod.example` без `BACKUP_*`.
- **Что не так:** off-site выполняется **только** если заданы `BACKUP_S3_URI`/`RCLONE`/`REMOTE_CMD` + ключ шифрования; иначе — бэкап **только локальный, на том же VPS**. Коммит ca9cd1d добавил *возможность* шифрованного off-site, но не его *включение*. Потеря сервера = потеря бэкапов — ровно сценарий соседа t2 (см. [[t2-block-risk-and-vanmark-isolation]]).
- **Рекомендация:** закоммитить образец `deploy/cron.d/vanmark-backup` (с `BACKUP_S3_URI` + `BACKUP_AGE_RECIPIENT`); добавить шаг установки в `deploy-release`; после настройки убедиться, что в логе появляется `off-site S3 ok`; приватный age-ключ хранить вне сервера; **отрепетировать восстановление** до пилота.
- **Уверенность:** высокая.

### В6. Нет security-заголовков (HSTS / X-Frame-Options / CSP / глобальный nosniff)
- **Где:** `Caddyfile` (только `reverse_proxy`), `next.config.ts` (нет `headers()`).
- **Что не так:** нет HSTS (downgrade/MITM на первом заходе при cookie-сессии), нет `X-Frame-Options`/CSP `frame-ancestors` (кликджекинг на кнопки смены статуса/расценки), нет глобального `nosniff`. *Поправка верификатора: раздача вложений (`/api/attachments/:id`) уже отдаёт `nosniff` — этот вектор закрыт; недостаёт глобальных заголовков.*
- **Рекомендация:** блок `header { Strict-Transport-Security "max-age=31536000; includeSubDomains"; X-Content-Type-Options nosniff; X-Frame-Options DENY; Referrer-Policy same-origin }` + базовый CSP в `Caddyfile`. HSTS включать после подтверждения, что весь трафик уже https.
- **Уверенность:** высокая.

### В7. `withIdempotency` (ядро офлайн exactly-once) не покрыт тестами
- **Где:** `src/domain/idempotency.ts` — нет `idempotency.test.ts`; e2e `offline.spec.ts` проверяет только накопление очереди и одиночную досылку, без дедуп-ассерта.
- **Что не так:** это единственная защита от задвоения DONE/фото/комментария при повторной офлайн-досылке. Регрессия (потеря проверки `prior.userId`, кэширование при ошибке) пройдёт зелёный CI и задвоит деньги/события в бою.
- **Рекомендация:** unit на `withIdempotency`: пустой ключ → `run()` вызван; новый → один раз + запись; существующий свой → `run()` не вызван, вернулся снимок; чужой ключ → 404; `run()` бросил → запись не создана; P2002 → не падаем. Плюс e2e: повтор `transition` с тем же `Idempotency-Key` → идентичный ответ и **ровно один** `TaskEvent`.
- **Уверенность:** высокая.

## 🟡 Мелочь / тех-долг (на потом)

**Безопасность/сессии:**
- `session.maxAge` не задан — JWT живёт дефолтные 30 дней без idle-таймаута/отзыва (`src/lib/auth.ts:27`).
- Заблокированная учётка отвечает заметно быстрее (троттлинг до argon2) — слабый timing-оракул на факт блокировки (`auth.ts:41`).
- MIME вложения берётся из client-controlled `file.type` без проверки сигнатуры (`attachment-service.ts:46`) — обойти «только фото» можно переименованием; раздача защищена `nosniff`+`inline`, прямого XSS нет.
- `/api/health` публичен — раскрывает факт сервиса и состояние БД (`health/route.ts`).
- `admin/pay-profiles` допускает **отрицательные** оклад/премию (валидируется только `Number.isFinite`) — `route.ts:25-31` *(firsthand)*.

**Деньги/смены/логика:**
- Гонка при закрытии месяца → 500 INTERNAL вместо `PERIOD_CLOSED` (`kpi-service.ts`).
- Исключение отпуска для «невыполненной точки» не применяется в лайв-перепроверке кандидатов (`kpi-service.ts`).
- Снятие требования акта (`requiresAct=false`) на DONE-задаче в открытом месяце меняет базу бонуса задним числом (`task-service.ts:421`).
- `assignTask`/`planTask` меняют `ASSIGNED↔NEW` не через `checkTransition` (ребра `ASSIGNED→NEW` в матрице нет) — намеренная «ось назначения», диспетчер-only *(firsthand + workflow)*.
- `confirmShift` без правки времени не пересчитывает `SHIFT_LATE` — поздняя смена, подтверждённая после ночного cron, теряет штраф (`shift-service.ts:193`).
- `SHIFT_REQUIRED` не привязан к текущему дню — незакрытая вчерашняя смена позволяет работать сегодня без открытия (`task-service.ts:659`).
- Офлайн-открытие смены пишет `openedAt = время досылки`, не реального открытия → возможен ложный `SHIFT_LATE`; лечится ручной корректировкой диспетчера (`adjustShiftOpenedAt`) *(firsthand)*.
- `periodBoundsUtc` хардкодит `+03:00`, тогда как `periodOf`/`dateKeyInTz` уважают `CRON_TZ` — разъедутся при смене зоны (`kpi.ts:61` vs `:47`) *(firsthand)*.
- FIFO-порядок досылки опирается на `seq=Date.now()` — ничья при двух действиях в одну мс (`offline/send.ts:50`).

**Прод/устойчивость/конфиг:**
- Образ `app` без версионного тега — откат только пересборкой, нет отката на предыдущий образ (`docker-compose.prod.yml:28`).
- `restart: unless-stopped` не перезапускает **unhealthy**-контейнер — упавший по health app/Postgres просто висит (`docker-compose.prod.yml`).
- Нет лимитов памяти/CPU и swap-защиты на 1.9 ГБ VPS (`docker-compose.prod.yml`).
- Раздача вложений грузит файл целиком в память (`Uint8Array`), без стриминга (`attachments/[id]/route.ts`).
- Логирование — 4 `console.error/warn` на весь сервер, без структуры (`api-route.ts`).
- `AUTH_SECRET` обязателен, но без дефолта и preflight-проверки — пустой → падение в рантайме на первой авторизации (`auth.ts:25`).
- `SEED_PASSWORD="vanmark123"` в `.env.example` — риск при копировании dev-шаблона на прод (`.env.example:12`).
- **Auth.js v5 в бете** (`next-auth@5.0.0-beta.31`) на проде — осознанный риск; апгрейд на стабильный релиз после запуска *(firsthand)*.

**Производительность/тесты:**
- `listTasks` для «Все задачи» грузит всю таблицу без пагинации и фильтра по дате (`task-service.ts`) — на старте безопасно (10-30 задач/день), но растёт.
- `setDriverSpecializations`: последовательные `updateMany` в цикле (`capacity-service.ts`).
- Частые выборки по `completedAt`/`worksheetStatus` без покрывающего индекса (`schema.prisma`).
- Нет unit на серверные инварианты жизненного цикла (`ACTIVE_TASK_EXISTS`, `SHIFT_REQUIRED`, гейт оплаты), на ограничение цены в `priceWorksheet`, на диапазон `lat/lng`.

**Зависимости:** `pnpm audit` — 8 advisories (1 high `hono` CORS, 6 moderate, 1 low), но **все только через dev/build-инструменты** (`@prisma/dev`→hono, `tsx`/`vitest`→esbuild, build-time `postcss`); в рантайм-путь прода не входят. Реальный риск низкий.

## 🔀 Расхождения: код ↔ документы / память

1. **`requiresPhoto` — устаревший комментарий схемы и правило №4 CLAUDE.md.** Схема (`schema.prisma:119`) и **жёсткое правило №4 CLAUDE.md** утверждают «DONE без фото запрещён — проверка на сервере». Фактически фото-гейт **снят** (этап 11): `Errors.photoRequired` нигде не вызывается, единственный гейт DONE — оплата ON_SITE. ARCHITECTURE §4/§5 и PRD §3/§5 уже описывают фото как НЕ-гейт, и `seed.ts:54` прямо пишет «поле больше не используется как гейт» — то есть **права сторона кода/PRD/ARCH**, а отстали комментарий схемы + CLAUDE.md. **Решение Артёма:** подтвердить смягчение → обновить правило №4 CLAUDE.md и комментарий схемы (пометить поле DEPRECATED, убрать неиспользуемый `Errors.photoRequired`). То же для «акта» в правиле №4 (`requiresSignedDoc` — мягкая KPI-метрика, не запрет).
2. **Расценка после подписания акта заблокирована** (`work-service.ts:279`): `priceWorksheet` принимает только PRICING/PRICED; после `SIGNED` цену не исправить (удалить акт при DONE тоже нельзя). Бизнес-логично, но в PRD §13 явно не зафиксировано — **подтвердить намеренность**.
3. **`REQUESTED`-смена не штрафуется за позднее открытие** (`kpi.ts:118`), хотя PRD §5.1 говорит «зачётное время = момент нажатия водителя независимо от подтверждения». Согласовать код с PRD.
4. **`seed-pilot.ts` ссылается на имена типов, которых нет в `seed.ts`** → `db:seed:pilot` падает на чистой репо-БД; на проде сработал только из-за исторического дрейфа справочника типов. Привести имена к каноническим.
