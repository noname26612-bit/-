# Деплой и релизы

Прод: один VPS в РФ, Docker Compose (`app` + `postgres` + `caddy`), фото в volume `/data/uploads`. Деплой допустим в нерабочее время команды (до 08:00 / после 20:00 МСК) или по согласованию с Артёмом — кроме хотфиксов.

## Первичная настройка сервера (один раз)

1. VPS: Ubuntu LTS, 2 vCPU / 2–4 ГБ; ufw: открыты только 22/80/443; ssh-ключи, root-вход выключен; fail2ban.
2. Docker + compose; склонировать репозиторий; заполнить `.env` по `.env.example` (секреты сгенерировать: `openssl rand -base64 32`, VAPID — `npx web-push generate-vapid-keys`).
3. DNS A-запись → IP; Caddy получит TLS автоматически.
4. Cron-бэкапы (см. ниже) + проверить восстановление на чистом контейнере ДО запуска пилота.
5. Сид: пользователи (Милена, водители, Артём-админ), справочник типов задач, стартовый номер sequence (уточнить у Артёма последний номер из Telegram-чата).

## Каждый релиз — по порядку

1. Локально: `pnpm lint && pnpm typecheck && pnpm test && pnpm e2e` — зелёные; security-check пройден, если менялись API/БД.
2. На сервере перед обновлением — бэкап: `pg_dump` в файл с тегом релиза + снапшот uploads (если менялась логика файлов).
3. `git pull` → `docker compose build app` → `docker compose run --rm app pnpm prisma migrate deploy` → `docker compose up -d app`.
4. Healthcheck: `curl -f https://<домен>/api/health` (эндпоинт обязателен с этапа 1).
5. Smoke на проде: вход диспетчером, открыть «Сегодня»; вход водителем с телефона, открыть «Мои задачи»; создать тестовую задачу и удалить.
6. Сообщить Артёму: версия, что вошло, что проверить руками.

## Миграции — правила безопасности

- Только `prisma migrate deploy` на проде (никогда `migrate dev` / `db push`).
- Разрушающие миграции (удаление/переименование колонок с данными) — в два релиза: сначала совместимый код, потом удаление. Перед ними — обязательный бэкап и подтверждение Артёма.

## Бэкапы (cron на VPS)

```
0 3 * * * pg_dump -Fc > /backups/db-$(date +\%F).dump      # ежедневно
30 3 * * * tar czf /backups/uploads-$(date +\%F).tgz /data/uploads
0 4 * * * find /backups -mtime +14 -delete                  # ротация 14 дней
```

Плюс копия последнего бэкапа на внешнее хранилище (S3-совместимый бакет или второй сервер). Раз в месяц — тестовое восстановление.

## Откат

1. `git checkout <предыдущий тег>` → build → up.
2. Если миграция несовместима — восстановить дамп: `pg_restore --clean` из бэкапа шага 2 (потеря данных с момента бэкапа — сообщить Артёму, восстановить вручную из журнала, если критично).
3. Записать причину отката и починить в ветке, не на проде.

## Запрещено

- Деплой без бэкапа. Ручные правки кода/БД на сервере. `docker compose down -v` (убивает volumes). Секреты в логи/коммиты.
